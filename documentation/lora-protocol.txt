================================================================================
ConeZ LoRa Protocol  --  v1 (clean-slate design)
================================================================================

A star-network RF link between one master node (Raspberry Pi + SX1262/SX1268)
and many ConeZ / Heltec cones (ESP32 + SX1262/SX1268). The master owns time and
the channel; cones find it, register, get polled, receive cues, and converge
their local `dist/` directory (and firmware) to match the master's.

This is the v1 protocol. There is no prior version to stay compatible with;
where this spec and the current code disagree, this spec is the target (see
Appendix B for what is actually built today).

Conventions: multi-byte integers are BIG-ENDIAN. "Cone" = any remote node
(ConeZ PCB with PSRAM, or Heltec without). Sizes/counts marked (tunable) are
master-chosen and advertised, not hard constants.


--------------------------------------------------------------------------------
1. Goals (what the link must do)
--------------------------------------------------------------------------------

  G1  Beacon so cones can find the master's channel + RF parameters.
  G2  Cones register with the master to get a polling slot (contention join).
  G3  Master polls registered cones for stats / data readback.
  G4  Master broadcasts cues to a cone, a group, or all (run wasm, set config,
      trigger, ...), with fire-now and fire-at-time variants.
  G5  Eventual-consistency file distribution: cones' `dist/` dirs and firmware
      converge to the master's, via spare-airtime broadcast (one-way in v1).
  G6  Master manages the timeslots (a scheduled superframe).
  G7  A scanlist guides cones on which channels/parameters to try.
  G8  Support FSK as well as LoRa modes (higher throughput when the link allows).


--------------------------------------------------------------------------------
2. Regulatory / operating context
--------------------------------------------------------------------------------

This system operates under US FCC Part 97 AMATEUR RADIO rules. The default
431.250 MHz channel is in the 70 cm amateur band (420-450 MHz), under the control
operator's amateur license.

  * STATION ID. The master's callsign rides in every BEACON (section 10); a
    cone's callsign rides in every uplink it sends (REG_REQ, POLL_RESP). A cone
    that only ever receives transmits nothing and needs no ID. This satisfies
    Part 97.119 (identify at least every 10 min of, and at the end of, a
    transmission) without putting a callsign in every packet.
  * NO ENCRYPTION. Part 97.113 forbids messages encoded to obscure meaning.
    Payloads stay in the clear. Data COMPRESSION is fine (it doesn't obscure
    meaning) -- hence the file format has a compression field but no cipher.
  * The callsign is per-control-operator, not a protocol constant.
  * Content is limited to what amateur rules permit (no business traffic, music).


--------------------------------------------------------------------------------
3. Topology, roles, addressing
--------------------------------------------------------------------------------

Single master, up to ~254 cones, ~5 mi range (master on a high/ridge antenna).
Half-duplex: only one station transmits at a time; the master schedules who.

  Node address (1 byte):
     0          master
     1 .. 254   a registered cone (assigned at registration)
     255        broadcast (all cones)

  Groups (1 byte group id, 1..254; 0 = "none"): a cone may belong to one or
  more groups (set by config or by a cue). Group-targeted cues are sent as radio
  broadcast (dst=255) and filtered by group id in the cue body (section 13).

  Network id (1 byte): lets two independent ConeZ networks share a band without
  cross-talk (in addition to the LoRa sync word). 0 = default network.


--------------------------------------------------------------------------------
4. Physical layer & channels
--------------------------------------------------------------------------------

The master sits on ONE channel and ONE mode (LoRa or FSK) at a time, advertised
in the beacon and discoverable via the scanlist (section 14). It may switch
channel/mode (e.g., drop to a fast FSK mode to push firmware, then return to a
robust LoRa control mode) -- always pre-announced in the beacon so cones follow.

  Control plane (always robust): beacon, registration, polling, cues are meant
  to run on a robust long-range LoRa mode so every cone, even distant ones, stays
  reachable. Default: 431.250 MHz, BW 500 kHz, SF7, CR 4/5, sync word 0xDEAD.

  LoRa params: frequency, bandwidth, spreading factor, coding rate, preamble.
     - LDRO (Low Data Rate Optimize) is COMPUTED, never hard-coded:
       ldro = (2**SF / BW_Hz) > 0.016 . TX and RX must agree (Appendix A).
  FSK params: bit rate, freq deviation, RX bandwidth, shaping, sync word, CRC.

  Airtime is the whole budget. Beacons + cues + the dist carousel + polling all
  share one half-duplex channel; at low SF / long range / large fleet airtime is
  the bottleneck, so the superframe (section 6) budgets it explicitly.

  Integrity/length come from the LoRa PHY (explicit header + hardware CRC), so
  the v1 application header carries neither.


--------------------------------------------------------------------------------
5. Frame format -- the lean common header
--------------------------------------------------------------------------------

Every v1 packet begins with a 4-byte header (vs the legacy 16-byte header). The
LoRa sync word + network id filter the network; the PHY gives length + CRC.

  Offset  Len  Field
  ------  ---  --------------------------------------------------------------
  0       1    Packet type (section 7)
  1       1    Network id
  2       1    Source addr   (0 = master)
  3       1    Dest addr     (0 = master, 1-254 = cone, 255 = broadcast)
  4       ..   Type-specific body

Callsign and magic are NOT in the header -- only the beacon and cone uplinks
carry a callsign, in their bodies (section 2).


--------------------------------------------------------------------------------
6. Superframe -- the master-scheduled time structure (G6)
--------------------------------------------------------------------------------

Time is divided into repeating superframes. The master starts each with a beacon
whose FRAME MAP describes the rest, then transmits in the downlink window, then
goes SILENT for the uplink windows so cones can talk.

   |<---------------------------- superframe (tunable) ------------------------>|
   [ BEACON ][ DOWNLINK: cues > dist carousel > poll requests ][ UPLINK: master silent ]
                                                               |  reg contention | poll slots |

  - BEACON: time, channel/params, frame map, manifest version, callsign.
  - DOWNLINK (master TX), priority order:
      1. Cues (lowest latency; fire-now repeated a few times).
      2. dist carousel pages (background; section 13).
      3. Poll requests (to the cone(s) scheduled this frame).
  - UPLINK (master RX, silent):
      * Registration contention window: K (tunable) slotted-Aloha sub-slots for
        unregistered cones (section 11).
      * Poll-response slot(s): the cone(s) polled this frame answer here
        (section 12). Poll rate is master-configurable, default as low as 1
        cone per superframe.

The beacon's frame map tells cones where the silent windows are, which slot is
the contention window, and which cone (if any) is being polled, so a cone only
transmits in a slot the master granted it. That single rule is timeslot
management (G6).


--------------------------------------------------------------------------------
7. Packet types
--------------------------------------------------------------------------------

  Type  Name         Dir         Purpose
  ----  -----------  ----------  ------------------------------------------------
  0x01  BEACON       M -> bcast  time, channel, frame map, manifest ver, callsign
  0x10  REG_REQ      C -> M      join request (in contention slot)
  0x11  REG_RESP     M -> C      assigned address + poll slot
  0x20  POLL_REQ     M -> C      request stats / config readback
  0x21  POLL_RESP    C -> M      stats / requested data (+ callsign)
  0x30  CUE          M -> *      run/trigger/set, fire-now or fire-at-time
  0x40  DIST_DATA    M -> bcast  a data chunk of a dist block
  0x41  DIST_PARITY  M -> bcast  an RS parity chunk of a dist block
  (0x50+ reserved: uplink gap-request, ACKed/critical cues, real-time FSK, ...)

The manifest is not a packet type -- it is a well-known dist file (section 13).


--------------------------------------------------------------------------------
8. Time synchronization (supports fire-at-time cues)
--------------------------------------------------------------------------------

  * ConeZ PCBs have GPS + PPS and keep their own precise epoch (firmware Time
    System); they do not need the beacon for time.
  * Non-GPS cones (Heltec) discipline their clock from the BEACON timestamp.
    Reference convention: the beacon's time field is the master's epoch at
    end-of-beacon-transmit; the cone stamps RX-done and corrects for the known
    air time. Good to ~tens of ms -- fine for fire-now and coarse fire-at-time.
  * The master clock is assumed good (Pi NTP- or GPS/PPS-disciplined).
  * Fire-at-time cues (section 13/G4) carry an absolute epoch; all cones execute
    at the same instant regardless of RX jitter. Precise sync is the GPS cones'
    job; non-GPS cones are as good as the beacon transfer.

--------------------------------------------------------------------------------
10. BEACON (0x01)   M -> broadcast (G1, G6, G7)
--------------------------------------------------------------------------------

Body (after the 4-byte header):
  - protocol version
  - master epoch: seconds (u32) + millis (u16)  [end-of-TX reference, section 8]
  - current channel descriptor: mode (LoRa/FSK), frequency, and the mode params
    (BW/SF/CR or bitrate/fdev/rxbw/shaping), sync word
  - frame map: superframe length, downlink length, uplink layout
      (contention window slot count + position, poll slot position),
      and which cone address (if any) is polled this frame
  - manifest serial (u16): current dist manifest version, so a cone knows if its
    dist/ dir is stale (section 13)
  - registration policy: contention slot count, backoff parameters
  - callsign (ASCII, e.g. 6 bytes space-padded) -- station ID (section 2)

Beacon cadence is tunable (the legacy 10 s is a fine starting point and easily
meets the 10-minute ID rule).


--------------------------------------------------------------------------------
11. Registration (G2)
--------------------------------------------------------------------------------

An unregistered cone, after locking to a beacon:
  1. Reads the contention window from the frame map.
  2. Picks a random sub-slot and sends REG_REQ in it:
       cone unique id (efuse/MAC, ~6 bytes), board/capability flags
       (PSRAM y/n, FSK y/n, ...), desired group(s), callsign.
  3. Master replies REG_RESP (next downlink): echoes the unique id, assigns a
     node address (1-254) and a poll-slot index.
  4. Collisions (two cones, same sub-slot) -> no REG_RESP -> exponential backoff
     and retry. Size the contention window for fleet-wide power-on (thundering
     herd is the main scaling risk here).

Re-registration recovers a dropped/forgotten assignment. The master keeps the
registry (address -> unique id, groups, poll slot, last-seen).


--------------------------------------------------------------------------------
12. Polling (G3)
--------------------------------------------------------------------------------

Master, at a configurable rate (default 1 cone/superframe, scaling down as the
fleet grows), round-robins POLL_REQ to registered cones; the polled cone answers
POLL_RESP in its uplink slot:
  - stats: last RSSI/SNR, rx/tx packet counts, firmware version, status flags,
    battery/solar voltage, temperature
  - or requested config readback (key list -> values)
  - callsign (station ID, section 2)
Stats are not time-critical, so slow polling is fine. No ACK needed.


--------------------------------------------------------------------------------
13. Cues (G4)
--------------------------------------------------------------------------------

Broadcast control, NO acknowledgements (a missed cue is acceptable).

CUE body:
  - cue sequence number (u16) -- lets a cone notice it missed one (e.g. via a
    later poll); not used for ACK/retransmit in v1
  - target: type (unicast / group / broadcast) + id (cone addr or group id);
    group/broadcast cues go out as radio broadcast and are filtered locally
  - execution: FIRE_NOW or FIRE_AT_TIME(epoch sec+ms)
      fire-now: low precision, repeated 2-3x for reliability
      fire-at-time: precise, executed at the carried epoch (section 8)
  - command: opcode + args, e.g.
      RUN     <path> <params>     run a wasm/bas/c program already on LittleFS
      STOP                        stop the running program
      SETCFG  <key> <value>       set a config parameter
      SETGRP  <group mask/op>     change group membership
      TRIGGER <id>                generic trigger
High-precision timing is always fire-at-time (cue files / wasm already resident
on LittleFS, delivered earlier via dist); real-time streaming is future (FSK).


--------------------------------------------------------------------------------
14. dist -- file & firmware distribution (G5)
--------------------------------------------------------------------------------

Goal: each cone's `dist/` directory (and its firmware) becomes eventually
consistent with the master's, via one-way broadcast in spare airtime -- the
master fills daytime/idle airtime so the night's show files (cue programs, wasm,
etc.) and firmware are present when needed. ("dist" replaces the misleading name
"rsync"; this is one-way push with eventual consistency, not bidirectional sync.)

13.1  THREE GRANULARITIES
  chunk  -- one on-air packet payload (<= ~240 B on LoRa's 255-byte cap; larger
            on FSK). Also one Reed-Solomon symbol.
  block  -- the compress + FEC + commit unit, ~32-64 KB (tunable). This is the
            key sizing decision (13.6).
  file   -- a manifest entry; = N blocks.

13.2  MANIFEST (the source of truth; drives fetch AND delete)
  A well-known high-priority dist file listing every file the master is serving.
  Per entry:
     file id (u16), path, uncompressed size, content hash,
     compression algo (0=none, deflate), block size, chunks/block (N),
     parity/block (R), total blocks, kind (data file | firmware | scanlist).
  The beacon advertises the manifest serial; when it changes, cones refetch the
  manifest, then: download files they lack or whose hash differs, and DELETE
  local dist files no longer in the manifest. Firmware is a manifest entry whose
  "kind" routes it to the OTA path (13.5).
  As BUILT (Phase 4): the manifest is tab-separated text with a [firmware] and a
  [files] section; the [files] line is
     id <tab> name <tab> uncompressed_size <tab> md5_8 <tab> algo <tab> block_size
     <tab> total_blocks
  (algo 0=store, 1=deflate). chunks/block (N) is NOT recorded -- it is variable
  with compression, so the cone learns it per block from the wire (total_chunks);
  parity/block (R) and an explicit "kind" arrive with Phases 5/6. The manifest
  file itself (id 0) is sent uncompressed at LP_DIST_BLOCK_SIZE.

13.3  ENCODING (master side)
  Per block: compress (deflate) the block's bytes -> split the COMPRESSED bytes
  into N data chunks -> append R systematic Reed-Solomon (GF(256)) parity chunks
  (N + R <= 255). Per-block (not per-file) compression costs only ~1-3% vs
  whole-file at >=32 KB blocks (measured on firmware.bin) while avoiding any
  whole-file buffering -- so it is essentially free here. Compress THEN FEC.
  As BUILT (Phase 5): the RS runs ACROSS chunks -- each chunk is one symbol
  column of length CHUNK_SIZE (the last data chunk is zero-padded to it; the wire
  carries block_comp_len so padding is unambiguous). The R parity rows are an
  R x N Cauchy matrix over GF(2^8) (poly 0x11D); systematic generator [I_N ; C].
  Cauchy guarantees every square submatrix is invertible, so the code is MDS and
  the cone can erasure-decode from ANY set of N survivors. R per block =
  max(DIST_PARITY_MIN, ceil(N*DIST_PARITY_FRAC)). Identical math in
  firmware/src/lora/rs.c and lora-master/reed_solomon.py.

13.4  RECEPTION (cone side)
  The carousel sends chunks in priority order (13.7), tagged file id + manifest
  serial + block index + chunk index. A cone:
    1. Buffers a block's chunks (one block needs <= ~50 KB for 64 KB blocks).
    2. With all N data chunks -> use them directly. With gaps but >= N of the
       N+R chunks -> Reed-Solomon ERASURE-recover the block (we know which chunks
       are missing from the PHY CRC, so erasure decode, not error correction).
    3. inflate_stream() the recovered compressed block straight to storage
       (LittleFS file at the block's offset; or flash for firmware, 13.5).
    4. Mark the block done in a per-file block bitmap; missing blocks are caught
       on the next carousel cycle (eventual consistency). No gap-requests in v1.

  FEC degrades gracefully by node class:
    - ConeZ (PSRAM): buffers freely, uses parity whenever it has gaps.
    - Heltec / no-PSRAM: it has RAM, just not megabytes. If it got all N data
      chunks of a block it commits and IGNORES the parity (systematic code -> no
      decode, no RAM cost). If it missed some it MAY RS-decode that one block
      (one block fits its RAM) or simply wait for the next carousel cycle.

13.5  FIRMWARE OTA (no PSRAM and no in-place hazard)
  Firmware blocks decompress into the INACTIVE app/OTA partition at their block
  offsets (erase sectors, write the inflate output as it streams). When the block
  bitmap is full, verify the whole-image hash from the manifest, then mark the
  OTA slot valid and reboot.
  NOTE on why NOT in-place: storing the whole compressed image in the partition
  and decompressing it in-place risks the decompress write-head overtaking the
  unread compressed read-head (solvable with a precomputed safety offset like the
  Linux self-extractor, but fragile -- a wrong margin bricks the image). The
  block model sidesteps it: the compressed block lives in RAM and inflate streams
  output to flash; we never decompress within the partition.
  As BUILT (Phase 6): firmware is matched by product (esp_app_get_description
  project_name) + version, downgrades refused unless [lora] ota_downgrade=on;
  per block the cone RS-recovers (if needed), inflates, and erase+writes the
  inactive OTA partition, then esp_ota_set_boot_partition (whole-image validate) +
  reboot. CRITICAL: a cone running normal firmware flashes its OTA partition WHILE
  the radio/system run. The S3 executes code XIP (straight from flash through a
  cache), so the chip cannot service a code fetch while it is mid-erase; the
  DEFAULT esp_partition erase/write therefore DISABLE the cache and cross-core-
  stall the other core for the whole op (parking it in an IRAM spin loop, else it
  faults fetching cached code). On the ConeZ board that park INTERMITTENTLY ran
  slow, stretching the cache-off window to seconds ->
  Interrupt-WDT reset or a Core-1 "cache disabled but cached memory accessed"
  panic, killing the multi-hour transfer. The fix is CONFIG_SPI_FLASH_AUTO_SUSPEND
  (our Winbond W25Q64 / id 0xEF4017 supports erase-suspend): flash ops then take a
  MUTEX instead of disabling the cache, so the cache never goes off and there is no
  cross-core park to fail. It must be on (with the matching bootloader) for dist
  firmware OTA to be reliable. Diagnosis ruled out DFS, light sleep, WiFi, the
  PSRAM driver, and concurrent app tasks; only the cache-disable window mattered.
  WHY ONLY OTA SURFACED IT (LittleFS writes use the SAME cache-disable path but
  almost never hang): the stall is a rare per-erase event, and OTA is the worst
  case on both axes -- ~300 back-to-back sector erases (vs a few sectors for a
  config/log save), fired while the radio/inflate/RS/log-ring keep the OTHER core
  hot (vs LittleFS writes at idle from the CLI, when the park is serviced
  instantly). OTA samples the rare event hundreds of times under peak load; sparse
  low-load LittleFS writes essentially never do. AUTO_SUSPEND changes the lock
  strategy for ALL flash ops, so LittleFS/config/log writes are hardened too.

13.6  BLOCK SIZE -- the one real knob
  Trades: compression (bigger = marginally better, diminishing past ~64 KB),
  FEC headroom (RS caps at 255 symbols/block, so SMALLER blocks leave room for
  MORE parity -- a 32 KB block ~110 data chunks leaves ~145 for parity; a 64 KB
  block ~210 data leaves ~45), and RAM (compressed block + ~43 KB inflate
  working set). Starting point: ~32-48 KB compressed cap -> generous parity and a
  comfortable Heltec footprint. Tunable per file kind; test with real files.

13.7  CAROUSEL = WEIGHTED SCHEDULER (interleave firmware with small files)
  Not a fixed round-robin. Each turn the master picks the next block/chunk by
  priority: manifest and scanlist highest, then cue/show files (small -> complete
  in minutes), then firmware at low weight filling leftover airtime (a full image
  is hours). A cue file dropped in at 18:00 lands well before the firmware that
  has trickled since noon.

13.8  CONCURRENT TRANSFERS (files + firmware at once)
  The carousel may interleave a new firmware image AND new files. A ConeZ with
  PSRAM keeps several block-reassembly contexts in flight and absorbs both at
  once. A Heltec / non-PSRAM node, under RAM pressure, may follow just ONE
  transfer at a time (e.g. firmware OR data files) and pick up the deferred one
  on later carousel cycles -- eventual consistency makes that safe.


--------------------------------------------------------------------------------
15. Channel discovery & scanlist (G7)
--------------------------------------------------------------------------------

A cone finds the master by trying channels from its scanlist until it hears a
beacon, then locks to the beacon's advertised params. Channels are organised in
priority TIERS, scanned round-robin and widened over time.

  * Sources, in priority order -- each becomes a tier when present:
      1. /dist/scanlist.txt -- delivered over the air via dist (master-managed)
      2. /scanlist.txt      -- local root override
      3. built-in DEFAULT   -- a few most-likely channels, ALWAYS present
      4. EXHAUSTIVE sweep   -- a virtual tier covering every LoRa freq x BW x SF
                               in band, computed on demand (not stored). The last
                               resort, so a lost cone eventually finds a master
                               on any channel given enough time.
  * The master may move channel or switch mode; it pre-announces in the beacon so
    locked cones follow, and rejoin via the scanlist if they lose it.

Scanlist file format (text; one channel per line, scan order = file order). The
first token selects the mode -- L = LoRa, F = FSK -- so a single plan covers both
(G8). All frequencies are in Hz and sync words in hex; '#' comments and blank
lines are ignored:

    L <freq_hz> <bw_hz> <sf> <cr> <sync_hex>
    F <freq_hz> <bitrate_bps> <freqdev_hz> <rxbw_hz> <sync_hex>

  e.g.  L 431250000 500000 7 5 0xDEAD
        F 431250000 4800 5000 9700 0x2D

Receive-only channels: suffix the mode token with X -- LX / FX -- to mark a
channel LISTEN-ONLY. The cone may lock to a master heard there but will NEVER
transmit on it -- e.g. GMRS / business / itinerant frequencies (462/467 MHz)
outside the amateur allocation, where we may listen but not key up:

    LX <freq_hz> <bw_hz> <sf> <cr> <sync_hex>                      # receive-only LoRa
    FX <freq_hz> <bitrate_bps> <freqdev_hz> <rxbw_hz> <sync_hex>   # receive-only FSK

The built-in default table marks channels the same way via LORA_RX / FSK_RX. A
receive-only lock skips registration/polling (it goes straight to a listen-only
state -- no uplink, no poll responses) but still acts on received packets (cues,
dist, beacons) as normal. The config key [lora] rx_only=on forces this listen-only
behaviour on EVERY channel, and `lora off` / [lora] enabled=off disables the radio
entirely. All transmit funnels through one gate (lora_tx_allowed), so a marked
channel, rx_only, or a disabled radio inhibits every uplink uniformly.

Within a file, put the most-likely (primary) channels first. The canonical
example lives in the master's dist/scanlist.txt.

Tiered escalation + round-robin: the cone starts on tier 0 (the highest-priority
source). After scan_passes fruitless sweeps of the PRIMARY tier it widens the
search by one tier, up to all tiers. The active tiers are then scanned
ROUND-ROBIN -- one entry from each per step, each tier keeping its own cursor --
so the small primary tier stays frequently scanned even while a huge backup tier
(e.g. the exhaustive sweep) is also being swept; a master returning to a main
channel is found quickly regardless of backup-list size. On a decoded beacon the
cone LOCKS; after a prolonged beacon loss it re-scans from tier 0.

Knobs (config [lora], read live): scan_dwell (seconds per channel, default 12 --
must exceed the beacon period to reliably catch a beacon); scan_passes (primary
sweeps before widening, default 5). CLI: `lora scan on|off`; `lora` status lists
the tiers. Implemented in firmware/src/lora/scan.cpp. The exhaustive tier uses
the default sync word (it cannot enumerate per-channel sync); file/built-in tiers
carry real sync words. FSK-combo exhaustive is a future addition (section 16).


--------------------------------------------------------------------------------
16. FSK mode (G8)
--------------------------------------------------------------------------------

FSK is offered as an alternative mode, time-shared exactly like LoRa: the master
is on either a LoRa or an FSK channel at any moment, and cones find it via the
scanlist. FSK trades range for throughput -- useful to push firmware fast, and
the intended carrier for future real-time cueing / DMX-ArtNet streaming. Same
framing, addressing, and packet types apply; only the PHY changes.


--------------------------------------------------------------------------------
17. Open items / future
--------------------------------------------------------------------------------

  * Uplink gap-requests (a cone asking for specific missing blocks in its poll
    slot) -- deliberately out of v1 (does not scale to a large fleet); cones
    catch misses on the next carousel cycle.
  * ACKed / critical cues; per-cue delivery confirmation via polling.
  * Real-time cueing / DMX / ArtNet streaming over FSK.
  * Cross-block / fountain FEC layered on top of the per-block RS, if the block
    tail proves too long in practice.


================================================================================
Appendix A. Master radio configuration -- hard-won notes (Waveshare hat + LoRaRF)
================================================================================

First successful master->cone link was brought up 2026-06-20; first cone->master
(ConeZ board TX) the same day. The master runs on a Waveshare "SX1262 XXXM
LoRaWAN/GNSS HAT" (433 MHz) with the bundled (Waveshare-example) LoRaRF Python
library. Non-obvious things that had to be right:

  * RF SWITCH: handled by DIO2 via setDio2RfSwitch(). TXEN/RXEN GPIOs left unused
    (txen=-1, rxen=-1), matching the known-working lora-aprs code. (The theory
    that TXEN=BCM6 had to be host-driven was a red herring.)
  * TCXO: do NOT enable DIO3 TCXO control on this hat -- adding it (1.8 V) HUNG
    the transmit (TxDone never fired). The cone (RadioLib) uses its own TCXO.
  * TX POWER / PA: LoRaRF setTxPower() silently no-ops for the SX1262 below
    +14 dBm (dead `else: return` -> SetPaConfig never issued -> no RF). Fix:
    set the PA from a per-power table ported from RadioLib 7.6.0 SX1262 paOptTable
    (set_tx_power() in conez-master.py); clean output -9..+22 dBm. Bug symptom:
    TX "completes" (TxDone fires) but nothing radiates.
  * LDRO -- THE CRC KILLER: compute from symbol time, never hard-code. The cone's
    RadioLib auto-computes LDRO; the master must agree or the payload de-maps and
    every packet fails CRC (-7) while the robust explicit header still decodes
    (length looks right). Rule: ldro = (2**SF / BW_Hz) > 0.016. (SF7/BW500 -> off;
    SF12/BW125 -> on.)
  * FREQUENCY ACCURACY: excellent -- ~308 Hz offset measured at 7 mi. No trim.
  * RX SATURATION: at ~1 m bench range a +22 dBm signal overloads the cone RX
    (no decode). For RF-quiet bench testing terminate the Pi in a 50 ohm dummy
    load (leakage ~ -80 dBm at the cone). ALWAYS sleep/standby the radio (PA off)
    before swapping antenna/load -- a keyed PA into an open load can be destroyed.

Cone firmware (firmware/src/lora/): 16-bit sync word (0xDEAD) set by inverting
RadioLib's setSyncWord(syncWord, controlBits) packing; SF default 7; readData()
failures log the RadioLib error code. CONE TX: `lora send <text>` -> lora_tx()
-> radio.transmit() -> re-arm RX, guarded by lora_tx_active so the loopTask RX
poll does not race it on the SPI bus (no radio mutex yet). The board's Ebyte
E22-400M22S (SX1268 + 22 dBm PA/LNA) drives its TXEN/RXEN T/R switch from DIO2
through an on-board N-channel MOSFET -- CONFIRMED to switch cleanly at 3.3 V
logic (ConeZ->Pi decoded clean at -71 dBm / SNR 13, +14 dBm board TX).


================================================================================
Appendix B. Implementation status (built vs. designed)
================================================================================

Built (see documentation/lora-v1-implementation-plan.txt for phases):
  * v1 framing -- 4-byte header [type,network,src,dst] + packet dispatch; v1
    BEACON (master emits, cone parses); non-GPS clock disciplined from the
    beacon (priority GPS > NTP > beacon). [Phase 1]
  * Scanlist & channel lock -- cone scans the scanlist (L/F format, section 15),
    locks on a beacon, re-scans on loss. Tiered (dist file / root file / built-in
    default / exhaustive virtual sweep) with round-robin scanning + widen-on-
    failure; scan_dwell + scan_passes config. firmware/src/lora/scan.cpp. [Phase 2]
  * dist core -- master broadcasts the manifest + data files as DIST_DATA chunks
    (round-robin carousel); cone reassembles, MD5-verifies vs the manifest,
    writes /dist/<name>, and deletes local files no longer listed. Firmware
    images skipped (Phase 6). [Phase 3]
  * dist compression -- transfer is now per-BLOCK; each block is independently
    deflated (master Python zlib, deflate-vs-store per file) and the cone
    inflates each completed block (inflate_buf, auto-detects zlib) into the file
    at its block offset. DIST_DATA body carries block_idx + total_blocks; the
    manifest [files] line carries algo + block_size + total_blocks. Block size is
    a master knob. [Phase 4]
  * dist FEC -- systematic Reed-Solomon over GF(2^8), ACROSS chunks (each chunk =
    one RS symbol column), generator [I_N ; C] with C an R x N Cauchy matrix (MDS).
    Per block the master sends N DATA (0x40) + R PARITY (0x41) chunks; a cone needs
    any N of the N+R, and since PHY CRC marks which are missing it does pure
    ERASURE decode -- recovering lost data from parity within the same carousel
    cycle. Same math in firmware/src/lora/rs.c and lora-master/reed_solomon.py. The
    DIST body added data_chunks N + parity_chunks R + block_comp_len (22 B). With
    all N data present the cone skips decode (Heltec-friendly). [Phase 5]
  * dist firmware OTA -- firmware is a dist entry (manifest [firmware] gains
    algo/block_size/total_blocks); the cone matches its own product/version
    (esp_app_get_description), refuses downgrades unless [lora] ota_downgrade=on,
    streams blocks straight to the inactive OTA partition (no whole-file buffer),
    verifies via esp_ota_set_boot_partition, and reboots into the new image. The
    manifest is broadcast after every beacon so a late-locking cone isn't stranded.
    REQUIRES CONFIG_SPI_FLASH_AUTO_SUSPEND (see 13.5 note). Full 1.2 MB OTA verified
    over RF end-to-end. [Phase 6]
  * Master: v1 beacon + dist carousel (lora-master/conez-master.py). The earlier
    ad-hoc 16-byte-header broadcast is retired (dead helpers remain dormant).
  * Cone: LoRa RX (lora_rx) + TX via `lora send` (lora_tx, guarded by
    lora_tx_active -- no radio mutex yet).
  * Bidirectional link verified both directions on the default LoRa params.
